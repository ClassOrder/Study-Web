Process와 Thread

전통적인 방식의 프로그래밍은 모종의 이유로 순차적인 프로그래밍에 집착하였습니다. 그 이유란 순차적인 프로그램은 프로세서를 하나만 이용하고, 동작은 항상 일정합니다. 이런 점으로 코드 전체를 대상으로 하는 테스트를 통해서 대부분의 버그를 발견할 수 있고, 발견된 버그를 잡아내기 수월한 편이었습니다. 하지만 Multi Thread는 일종의 경량 Process로서, 같은 프로세스 내의 다른 스레드와 메모리를 공유합니다. Thread는 여러 개의 동시 Thread 실행을 수용할 수 있게끔 기존 모델을 확장하기 위해 개발되었습니다. 하나의 Thread가 I/O 작업을 기다리는 동안 다른 Thread는 CPU를 점유할 수 있습니다. 이런 특성 상 동작은 항상 일정하지 않고(부정) 코드의 범위만으로 테스트를 하는 것만으로는 불충분합니다. Multi Thread를 이용하는 경우에는 메모리에 접근하면서 Thread끼리 경합(충돌)하는 문제가 생긴다면 해결하기 어려운 문제가 발생할 수도 있습니다. 이런 경합에 의해 발생하는 버그의 경우 코드를 논리적으로 바라보고 원인을 찾아야 해결이 가능하기 때문에 근본 원인을 특정하지 못하거나 확인하기 어려운 경우가 발생합니다.
